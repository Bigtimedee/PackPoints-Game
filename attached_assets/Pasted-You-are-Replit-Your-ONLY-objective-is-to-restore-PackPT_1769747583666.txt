STEP 5 — SET NAME RESOLUTION (critical for “1987 Topps Baseball”)
CardHedge set filter is likely exact-match. If PackPTS uses friendly labels, it will miss.

Implement a set resolver layer:
Create: /server/integrations/cardhedge/setResolver.ts
- Input: (displaySetName, category)
- Output: resolvedSetName (exact string that CardHedge expects)

Algorithm:
1) Try exact match using:
   POST /v1/cards/card-search with { set: displaySetName, category, page:1, page_size:1 }
   - if count > 0 => resolvedSetName = displaySetName
2) If count == 0:
   - Run a fuzzy “search” using keywords from displaySetName:
     { search: displaySetName, category, page:1, page_size:50 }
   - From results, compute best candidate set string:
     - choose the most frequent `set` among results
     - prefer sets whose name contains the year and brand tokens (e.g., “1987”, “Topps”, “Baseball”)
   - Re-test that candidate via exact set query; if count > 0 use it.
3) Cache mapping in DB table:
   set_aliases(display_name, category, resolved_set, last_verified_at, count_snapshot)
4) On subsequent calls, use cached resolved_set, but re-verify weekly or if count suddenly drops to 0.

This is how you prevent “1987 Topps baseball (300 cards)” when reality is much higher.
