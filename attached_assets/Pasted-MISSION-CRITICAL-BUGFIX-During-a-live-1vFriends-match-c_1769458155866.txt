WHAT THIS ERROR MEANS
The client attempted to submit, the server rejected with something like:
- match.status != 'ACTIVE' (e.g., 'INITIALIZING', 'LOBBY', 'CANCELLED', 'FINISHED')
OR
- match not found / stale matchId
OR
- idx mismatch (server current_index != client idx) and you mapped that to “not active”
OR
- server restarted and lost in-memory match state (if any), so it thinks match is inactive
OR
- ACK/timeout logic cancelled the match even though gameplay started

GOAL
1) Server is the source of truth and ONLY rejects submit when:
   - user not participant, OR
   - match truly CANCELLED/FINISHED, OR
   - idx is stale (but that is NOT “not active”; it is “stale_index” + resync)
2) Client error messages must be specific:
   - “Match ended” only when status is FINISHED/CANCELLED
   - “Out of sync. Tap Resync.” for stale_index or status INITIALIZING
3) Eliminate any timer/ACK logic that cancels after gameplay has begun.
4) Persist match state in DB (not memory) and compute everything from DB.

B) BACKEND: FIX MATCH STATUS GATING

Implement single function:
src/services/matches/guardCanSubmit.ts

Inputs: match row, userId, idx
Logic:
1) if !match => reject match_not_found
2) if user not participant => reject not_participant
3) if match.status == 'CANCELLED' => reject match_cancelled
4) if match.status == 'FINISHED' => reject match_finished

5) if match.status == 'INITIALIZING':
   - Allow submit ONLY if match_questions exist AND idx == match.current_index AND idx==0
   - Otherwise reject match_initializing with serverIndex/status
   (This prevents false “not active” during the brief init window.)

6) if match.status == 'ACTIVE':
   - if idx != match.current_index => reject stale_index with serverIndex/status
   - else OK

IMPORTANT:
- Do NOT treat idx mismatch as “not active”. It is “stale_index”.
