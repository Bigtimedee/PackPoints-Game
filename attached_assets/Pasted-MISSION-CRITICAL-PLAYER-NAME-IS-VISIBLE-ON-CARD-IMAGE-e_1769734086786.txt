MISSION CRITICAL: PLAYER NAME IS VISIBLE ON CARD IMAGE (e.g., “JIM RICE”).
PackPTS must ALWAYS mask the player’s name on the CARD IMAGE before rendering it in gameplay.

Implement a robust, “intelligent masking” pipeline that supports:
- names printed at the TOP (banner/nameplate)
- names printed at the BOTTOM (name strip)
- occasional side/diagonal text
- and avoids relying on a single hard-coded crop.

DO NOT make a quick CSS overlay. This must be image-level masking (server-side), cached, and deterministic.

======================================================================
GOAL
For every card shown in gameplay, the client must only receive a “masked image URL” generated by PackPTS.
Never use the raw Card Hedge image URL directly in the UI.

======================================================================
A) BACKEND IMAGE MASKING SERVICE (SERVER-SIDE)
1) Create endpoint:
   GET /api/cards/:cardId/masked-image
   - Requires the card_id (Card Hedge) and uses our stored metadata (player name, set, raw image_url).
   - Returns: { masked_image_url }

2) Create storage/cache:
   - Save masked images to S3 (or your existing storage) under:
     /masked-cards/{cardId}/{maskVersion}.jpg
   - Add DB table to cache:
     card_image_mask_cache:
       card_id TEXT PK
       raw_image_url TEXT
       masked_image_url TEXT
       mask_version TEXT
       updated_at TIMESTAMP

3) The endpoint should:
   - If cached and raw_image_url unchanged AND mask_version current -> return cached masked_image_url
   - Else:
     a) download the raw image server-side
     b) run the masking pipeline (below)
     c) upload masked image
     d) update cache
     e) return masked_image_url

Use `sharp` for all image processing.

======================================================================
B) INTELLIGENT MASKING PIPELINE (TWO-LAYER DEFENSE)
We will implement BOTH:
(1) Template masks (fast, always-on)
(2) OCR-based targeted masks (smarter, token-aware)

This guarantees masking even if OCR fails.

----------------------------
B1) TEMPLATE MASKS (ALWAYS APPLY)
For every card image:
- Apply a semi-opaque blur/cover band at BOTH:
  - TOP band: y=0%..18% of height
  - BOTTOM band: y=78%..100% of height
These are typical nameplate regions across many sets, including cases like “JIM RICE” at top.

Implementation details:
- Use `sharp` to composite filled rectangles (e.g., dark/blurred overlay) over those regions.
- Use a slightly feathered edge if available (optional).
- Keep these percentages configurable by set via “set mask profiles”.

Create a config file:
  /server/masking/maskProfiles.ts

Example:
- default:
  topBandPct: 0.18
  bottomBandPct: 0.22
  leftBandPct: 0.0
  rightBandPct: 0.0
- "1987 Topps":
  topBandPct: 0.20
  bottomBandPct: 0.25

Important:
If we don’t recognize the set, use default (still masks top+bottom).

----------------------------
B2) OCR TARGETED MASKING (APPLY ON TOP OF TEMPLATE MASKS)
Add an OCR step that detects if player name tokens appear anywhere else (or to tighten/expand coverage):
- Use `tesseract.js` (Node) OR a lightweight OCR library that returns bounding boxes.
- Run OCR on a DOWN-SCALED version (e.g., width 700px) to keep it fast.
- Extract bounding boxes for detected words/lines.
- Compare detected text to the card’s player name tokens:
  tokens = ["jim", "rice"] (lowercased; include nickname variants if stored)
- If OCR detects any token match:
  - Expand bounding box padding by ~12–20px (scaled to image size)
  - Composite an additional mask rectangle over the matched region

If OCR fails or times out (e.g., > 2.5s):
- Proceed with template masks only (still safe).

Add a strict allowlist of characters and normalization:
- Lowercase
- strip punctuation
- normalize whitespace
- handle common OCR confusion (I/1, O/0) with fuzzy match (Levenshtein distance <=1 for short tokens)
