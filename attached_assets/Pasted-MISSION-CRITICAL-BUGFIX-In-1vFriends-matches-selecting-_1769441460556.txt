MISSION-CRITICAL BUGFIX: In 1vFriends matches, selecting an answer triggers an error and the user cannot submit or advance to the next question. Implement a reliable “select → submit → server validate → advance” flow with clear WS message contracts, server-side idempotency, and a fallback REST submit route. Ensure the UI always shows a Submit button and advances only after server ACK.

STACK (do not change)
Frontend: React 18 + Vite + TypeScript + Tailwind + shadcn/ui + TanStack Query + Wouter
Backend: Express + TypeScript
Realtime: ws
DB: Postgres (Neon) + Drizzle ORM
Auth: Passport + express-session

SYMPTOM
- User taps an answer option in 1vFriends
- App shows an error (likely unhandled WS send, missing matchId/idx, schema mismatch, or server rejects payload)
- No Submit button / no transition to next question

PRIMARY CAUSES (LIKELY)
A) The client is firing “submit” on click, but payload is missing required fields (matchId, idx, choiceId), so server rejects.
B) Client expects REST but server expects WS (or vice versa), causing 404/500 or ws “not connected”.
C) Server requires idx === current_index, but client is sending wrong idx (string vs number, off-by-one, undefined).
D) Server returns an error response, but client doesn’t handle it and locks UI.
E) No ACK protocol: client never receives “ANSWER_ACCEPTED”, so it never advances.

GOAL
- Clicking selects only (no network call).
- A visible Submit button sends answer with required fields.
- Server replies with explicit ACK (accepted/rejected) and then broadcasts NEXT_QUESTION when both players answered (or timer expires).
- UI advances on server events; never gets stuck.
- If WS fails, client can submit via REST fallback.

========================================================
A) MESSAGE CONTRACTS (MUST IMPLEMENT EXACTLY)

Client -> Server (WS)
1) ANSWER_SELECT (optional; can omit)
{ type: "ANSWER_SELECT", matchId: string, idx: number, selected: string }

2) ANSWER_SUBMIT (required)
{ type: "ANSWER_SUBMIT", matchId: string, idx: number, selected: string, clientMsgId: string }

3) MATCH_RESYNC
{ type: "MATCH_RESYNC", matchId: string }

Server -> Client (WS)
1) ANSWER_ACK (direct to submitting client)
{ type: "ANSWER_ACK", matchId: string, idx: number, clientMsgId: string, status: "ACCEPTED"|"REJECTED", reason?: string }

2) QUESTION (broadcast to both when advancing)
{ type: "QUESTION", matchId: string, idx: number, question: { cardImageUrl: string, choices: string[], meta?: any } }

3) MATCH_STATE (resync response)
{ type: "MATCH_STATE", matchId: string, status: "INITIALIZING"|"ACTIVE"|"FINISHED"|"CANCELLED", currentIndex: number, questionsTotal: number, question?: ... }

========================================================
B) BACKEND: WS HANDLERS (ROBUST + IDEMPOTENT)

Create:
src/ws/handlers/answers.ts

On ANSWER_SUBMIT:
1) Validate schema with Zod:
- matchId string
- idx number
- selected non-empty string
- clientMsgId string

2) Load match from DB:
- must exist
- must be status ACTIVE (or allow INITIALIZING if you want)
- must have idx === matches.current_index

If invalid:
- send ANSWER_ACK REJECTED with reason:
  "bad_payload" | "match_not_found" | "match_not_active" | "stale_index" | "not_participant"
Return.

3) Idempotency:
- Create table match_answer_events:
  - match_id uuid
  - user_id uuid
  - idx int
  - client_msg_id text
  - selected text
  - created_at timestamp
  Unique(match_id, user_id, idx)          // only one answer per idx per user
  Unique(match_id, user_id, client_msg_id)

If row already exists for (match_id,user_id,idx):
- treat as accepted and re-send ACK ACCEPTED
(do NOT error)

4) Insert answer row, compute correctness server-side (compare to match_questions.correct_player OR correct_id).
5) Send ANSWER_ACK ACCEPTED to submitting client immediately.

6) Determine if both players have answered for this idx:
- Count answers for match_id and idx for both participants
If yes:
- advance index:
  newIdx = idx + 1
  update matches.current_index = newIdx
  if newIdx >= questions_total:
     finish match and broadcast MATCH_END
  else:
     load match_questions for newIdx
     broadcast QUESTION to BOTH players

Add:
POST /api/matches/:matchId/answer
Body: { idx: number, selected: string, clientMsgId: string }

This should call the exact same service function used by WS handler and return:
{ ok: true } or { ok:false, reason }

Frontend uses this only if WS send fails or WS disconnected.

========================================================
D) FRONTEND: UI FLOW (SELECT THEN SUBMIT)

In the question view component:
State:
- selectedChoice: string | null
- submitting: boolean
- errorMsg: string | null

Behavior:
1) On option click:
   - setSelectedChoice(choice)
   - clear errorMsg
   - DO NOT submit automatically

2) Always show a Submit button:
   - disabled if selectedChoice is null or submitting

3) On Submit click:
   - submitting=true
   - generate clientMsgId = crypto.randomUUID()
   - send WS ANSWER_SUBMIT { matchId, idx, selected, clientMsgId }
   - start 3s timeout; if no ANSWER_ACK received, call REST fallback
   - On ANSWER_ACK ACCEPTED:
        show “Locked in” state and disable further changes
        wait for QUESTION broadcast to advance
   - On ANSWER_ACK REJECTED:
        show errorMsg = reason mapped to friendly text
        submitting=false (allow resubmit)

4) On QUESTION event:
   - update idx/question payload
   - setSelectedChoice(null)
   - submitting=false
   - errorMsg=null

5) Add “Resync” button (or auto on error):
   - ws send MATCH_RESYNC if stuck
   - server replies MATCH_STATE with current question
