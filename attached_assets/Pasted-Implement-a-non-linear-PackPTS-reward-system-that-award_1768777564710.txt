Implement a non-linear PackPTS reward system that awards fewer points for famous players and more points for obscure players, bounded between 100 (famous) and 200 (non-famous). This must be dynamic, tunable without redeploy, and integrated into gameplay scoring.

STACK (do not change)
Frontend: React 18 + Vite + TypeScript + Tailwind + shadcn/ui + TanStack React Query + Wouter
Backend: Express + TypeScript (tsx dev)
DB: Postgres (Neon) + Drizzle ORM + migrations
Auth: Passport + express-session
Realtime: ws (1v1 games)
Card data: CardHedge integration (cards + images)

BUSINESS REQUIREMENTS
- Reward range: MIN_PTS = 100, MAX_PTS = 200
- Famous player should earn ~100; obscure player should earn ~200.
- Use a continuous, non-linear curve (not if/else).
- Must support admin tuning (curve steepness, weights, multipliers) via DB-configurable policy.
- Must be safe against farming / inflation:
  - clamp rewards to bounds
  - optional daily cap and per-match cap
  - log every awarded score with inputs for auditability

DEFINITIONS
- fame_score f ∈ [0,1]
  - f=1 means very famous (lowest reward)
  - f=0 means obscure (highest reward)

BASE REWARD FORMULA (MUST IMPLEMENT EXACTLY)
Let:
minPts = policy.minPts (default 100)
maxPts = policy.maxPts (default 200)
gamma = policy.gamma (default 2.0)
f = fame_score

basePts = minPts + (maxPts - minPts) * (1 - (f ^ gamma))

Then apply optional modifiers:
finalPts = round(basePts * vintageMultiplier * rarityMultiplier)

Clamp:
finalPts = min(max(finalPts, minPts), policy.maxAwardCap)   // maxAwardCap default 250

BACKEND IMPLEMENTATION (Express)
1) Reward engine module
Create src/services/rewards/rewardEngine.ts:
- loadActivePolicy(): reads latest enabled policy by effective_from
- normalizePlayerKey(name, sport): lowercases, strips punctuation/spaces consistently
- getFameScore(playerKey): fetch player_fame; if missing create placeholder with default fame_score=0.5 and queue update
- getVintageMultiplier(year): uses policy JSON bucket logic
- getRarityMultiplier(rarityType): uses policy JSON
- computeBasePts(f, policy): EXACT formula above
- computeFinalPts(basePts, multipliers, policy): rounding + clamp

2) Fame score sources (Phase 1 minimal)
Implement a daily job stub that refreshes fame_score from available sources:
- If CardHedge provides a popularity/rank field for players/cards, use it
- Otherwise default to internal signals later
For now:
- Start with heuristic: if player appears frequently in our gameplay and has high correct rate, set fame_score higher.
Create table internal_player_stats:
- player_key, attempts, correct, updated_at

Job: recompute fame_score nightly:
- fame_internal = clamp(correct_rate_weighted, 0..1)
- fame_score = fame_internal initially
Store source_breakdown accordingly.

3) Award points when a user answers correctly
Wherever correct answers are processed (HTTP or ws message handler):
- Build player_key from card metadata
- Compute finalPts via rewardEngine
- Enforce caps:
  - daily cap: do not award beyond policy.daily_points_cap; clamp remaining
  - per-match cap: track awarded in match context and clamp
- Write to points_awards audit row with full snapshot
- Credit PackPTS to user wallet via existing append-only ledger:
  - ledger entry type=EARN, amount=finalPts, metadata includes points_awards.id
  - idempotency key: "award:<matchId>:<questionId>:<userId>"
- Return awarded amount to client.
