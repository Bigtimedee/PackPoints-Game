Implement “1vRandom Matchmaking” for PackPTS using WebSockets with a presence system + matchmaking queue + heartbeat validation + timeouts + fallback. This must reliably pair 2 active users into a 1v1 match only when BOTH are online and explicitly searching.

STACK (do not change)
Backend: Express + TypeScript + ws
DB: Postgres (Neon) + Drizzle ORM
Auth: Passport + express-session
Frontend: React 18 + Vite + TS + TanStack Query + Wouter

GOALS
- Maintain real-time presence of connected users
- Allow users to enter a 1vRandom matchmaking queue
- Pair two eligible users immediately when available
- Ensure users in queue are truly active (heartbeat + open socket)
- Expire stale queue entries
- Provide matchmaking status updates to client
- Provide fallback options when queue is empty or slow

========================================================
A) DATA MODEL (Drizzle migrations)

1) user_presence
- user_id uuid PK
- socket_id text
- status enum('ONLINE','OFFLINE','IN_MATCH','SEARCHING') default 'OFFLINE'
- last_seen_at timestamp not null
- updated_at timestamp default now

2) matchmaking_tickets
- id uuid PK
- user_id uuid not null
- mode enum('1vRandom') not null
- bucket text not null           // e.g. "Baseball|1987 Topps" or "ANY"
- status enum('WAITING','MATCHED','CANCELLED','EXPIRED') default 'WAITING'
- created_at timestamp default now
- updated_at timestamp default now
Unique(user_id, mode) WHERE status IN ('WAITING','MATCHED')  // enforce one active ticket

3) matches
- id uuid PK
- mode enum('1vRandom') not null
- bucket text not null
- player1_id uuid not null
- player2_id uuid not null
- status enum('CREATED','ACTIVE','FINISHED','CANCELLED') default 'CREATED'
- created_at timestamp default now
- started_at timestamp nullable
- finished_at timestamp nullable

========================================================
B) WEBSOCKET PROTOCOL

Create WS server:
src/ws/server.ts

On connect:
- authenticate session cookie -> userId
- create socketId
- upsert user_presence: ONLINE, last_seen_at=now

Heartbeat:
Client sends every 15s:
{ type: "PING" }
Server responds:
{ type: "PONG" }
Server updates user_presence.last_seen_at

On disconnect:
- set user_presence.status=OFFLINE
- cancel any WAITING matchmaking ticket for that user

========================================================
C) MATCHMAKING QUEUE (IN MEMORY + DB SAFETY)

Implement service:
src/services/matchmaking/queue.ts

Maintain in-memory maps:
- socketsByUserId: Map<userId, ws>
- waitingByBucket: Map<bucket, Deque<ticketId>>

But DB remains source of truth for recovery:
- tickets are persisted in matchmaking_tickets.

Bucket rules:
- If user chooses set: bucket = `${sport}|${setId}`
- Else bucket = "ANY"
Also allow “bucket widening” after timeouts (client prompt).

========================================================
D) ENTRYPOINTS

WS messages:

1) START_SEARCH
Client -> Server:
{ type: "START_SEARCH", mode: "1vRandom", bucket: "ANY" | "<sport>|<setId>" }

Server:
- Verify user is ONLINE and not IN_MATCH
- Create or replace ticket:
  - cancel prior WAITING ticket
  - insert matchmaking_tickets WAITING
- update presence status=SEARCHING
- enqueue ticketId into waitingByBucket[bucket]
- immediately attemptMatch(bucket)
- respond:
{ type: "SEARCH_STARTED", ticketId, bucket }

2) CANCEL_SEARCH
Client -> Server:
{ type: "CANCEL_SEARCH", ticketId }

Server:
- mark ticket CANCELLED
- remove from queue (best-effort)
- update presence status=ONLINE
- respond { type: "SEARCH_CANCELLED" }

========================================================
E) MATCH CREATION LOGIC (CORE)

Implement:
src/services/matchmaking/matchmaker.ts

function attemptMatch(bucket):
- While queue has >=2:
  - pop ticketA, ticketB (must be different users)
  - validateTicket(ticketX):
      - ticket.status == WAITING
      - user_presence.status == SEARCHING
      - last_seen_at within 30s
      - socket exists and open
      - not already in match
    If invalid: mark ticket EXPIRED and continue
  - create match row with both users
  - update tickets to MATCHED
  - update both user_presence.status=IN_MATCH
  - notify both sockets:
      { type: "MATCH_FOUND", matchId, opponent: { userId }, bucket }
  - transition match to ACTIVE after both ACK (see below)

ACK HANDSHAKE (prevents ghost matches):
- After MATCH_FOUND, each client must respond within 10s:
  { type: "MATCH_ACK", matchId }
- Server tracks acks (in memory)
- If both ack:
  - update matches.status=ACTIVE, started_at=now
  - send { type: "MATCH_START", matchId }
- If one fails:
  - cancel match
  - put the acked user back into SEARCHING with a new ticket
  - mark no-show user ticket EXPIRED and optionally apply cooldown

========================================================
F) STALE CLEANUP JOB

Implement interval job every 10 seconds:
- Expire tickets older than 90 seconds that are still WAITING
- If expired: set ticket EXPIRED and set presence ONLINE if still connected and not searching
- Also mark presence OFFLINE if last_seen_at older than 60s (missed heartbeats)

- Ensure “IN_MATCH” users cannot re-queue
