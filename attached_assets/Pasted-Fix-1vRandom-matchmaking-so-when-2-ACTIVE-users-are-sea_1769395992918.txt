Fix 1vRandom matchmaking so when 2+ ACTIVE users are searching, the server pairs them immediately and starts a match. Current bug: both users see “position #1” and “players in queue: 1” simultaneously, meaning each client is maintaining an isolated/local queue state instead of a single server-authoritative queue.

STACK (do not change)
Backend: Express + TypeScript + ws
DB: Postgres (Neon) + Drizzle ORM
Auth: Passport + express-session
Frontend: React 18 + Vite + TS + TanStack Query + Wouter

PRIMARY DIAGNOSIS
Matchmaking state is not centralized. Each websocket node/client is computing queue position from local memory or a per-socket list, causing both users to believe they’re alone (#1). The matchmaker must be server-authoritative using ONE shared queue + atomic pairing. If you run multiple server instances, you must use DB (or Redis) as the shared queue.

GOAL
- A single source of truth for the 1vRandom queue
- Deterministic pairing when at least 2 eligible tickets exist
- Accurate queue counts/positions (server-calculated)
- Handshake/ACK to prevent ghost matches
- Robust against disconnects and stale tickets

========================================================
D) ATOMIC PAIRING (THE FIX)
Create:
src/services/matchmaking/matchmaker.ts

Implement function:
attemptPair(mode='1vRandom', bucket='ANY')

Use a single DB transaction with row locking:

PSEUDOCODE (MUST IMPLEMENT WITH SQL/Drizzle):
BEGIN;

-- 1) Select the two oldest WAITING tickets that are alive
-- alive = last_heartbeat_at > now() - interval '30 seconds'
SELECT id, user_id
FROM matchmaking_tickets
WHERE mode=$mode AND bucket=$bucket AND status='WAITING'
  AND last_heartbeat_at > now() - interval '30 seconds'
ORDER BY created_at ASC
FOR UPDATE SKIP LOCKED
LIMIT 2;

IF less than 2 rows -> COMMIT and return (no match)

-- 2) Create match
INSERT INTO matches (...) VALUES (...) RETURNING id;

-- 3) Mark both tickets MATCHED
UPDATE matchmaking_tickets
SET status='MATCHED', updated_at=now()
WHERE id IN (ticket1, ticket2);

COMMIT;

-- 4) Notify both users via websocket:
send MATCH_FOUND to both sockets

IMPORTANT:
- Use FOR UPDATE SKIP LOCKED to prevent two concurrent pairers from grabbing the same users.
- This is the canonical fix for “both are #1”.

E) QUEUE POSITION + COUNT (SERVER-CALCULATED)
When a user starts searching, and every ~3 seconds while searching:
- Compute:
playersInQueue = COUNT(*) of WAITING tickets (alive) in same bucket
yourPosition = 1 + COUNT(*) of WAITING tickets (alive) with created_at < your_ticket.created_at
- Send SEARCH_STATUS to that user

This ensures both users see consistent numbers.

Implement:
src/services/matchmaking/status.ts
- getQueueStats(ticketId): {playersInQueue, yourPosition}

========================================================
F) START_SEARCH HANDLER (FIXED)
On START_SEARCH:
1) Upsert/replace user’s existing ticket:
- Cancel any prior WAITING ticket for this user+mode
- Insert new ticket with status WAITING, socket_id=current, last_heartbeat_at=now
2) Immediately call attemptPair(mode,bucket)
3) Send SEARCH_STATUS to the user

========================================================
G) HEARTBEAT + STALE CLEANUP
On HEARTBEAT:
- Update matchmaking_tickets.last_heartbeat_at=now WHERE user_id=? AND status='WAITING'
If no WAITING ticket exists, ignore.

Cleanup job every 10s:
- Mark tickets EXPIRED where status='WAITING' and last_heartbeat_at < now()-30s
- If a socket disconnects:
  - CANCEL any WAITING ticket for that user

========================================================
H) ACK HANDSHAKE (PREVENT GHOST MATCHES)
After MATCH_FOUND:
- Require each client to send MATCH_ACK within 10s
- When both ACK:
  - update matches.status='ACTIVE', started_at=now
  - send MATCH_START to both clients
- If one fails:
  - cancel match
  - set the ACKed user back to WAITING with a new ticket (same bucket)
  - mark no-show ticket EXPIRED and optionally impose cooldown

Implement either:
- match_acks table (recommended for reliability) OR
- in-memory ack tracking keyed by matchId with a 10s timer
