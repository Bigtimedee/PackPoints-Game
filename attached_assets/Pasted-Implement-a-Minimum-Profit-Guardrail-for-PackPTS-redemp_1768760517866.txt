Implement a “Minimum Profit Guardrail” for PackPTS redemptions applied to external marketplace purchases (eBay/Goldin). The system must compute the maximum PackPTS a user can redeem for a given card price while guaranteeing PackPts Inc. achieves a minimum profit margin on every transaction.

CONTEXT
- PackPTS is a rewards currency. Users can redeem PackPTS as “rebate credit” or “store credit” applied to purchases.
- We must enforce a mandatory minimum profit margin m on each transaction.
- Stack: React 18 + Vite + TS + Tailwind + shadcn/ui + React Query + Wouter; Express + TS; Postgres (Neon) + Drizzle; Passport + express-session.
- PackPTS uses an append-only ledger + buckets (FIFO) with expiration.

BUSINESS MATH (MUST IMPLEMENT EXACTLY)
Let:
P = purchase price in USD (from listing)
A = affiliate revenue rate (decimal, e.g., 0.02)
h = affiliate reliability haircut (decimal, e.g., 0.70)
m = minimum profit margin (decimal, e.g., 0.25)
r = processing fee rate applied to credit cost (decimal; if none, 0)
f = fixed fee per transaction in USD (default 0)
v = USD value per PackPTS (e.g., 0.002 => 1 PackPTS = $0.002)

Profit = (h*A*P) - C*(1+r) - f
Constraint: Profit >= m*P
Solve:
C_max = ((h*A - m)*P - f) / (1+r)
R_max = floor(C_max / v)

Rules:
- If C_max <= 0 then R_max = 0 (no redemption allowed for this purchase).
- Redemption must never exceed user’s available PackPTS balance.
- Always record the computed parameters (P, A, h, m, r, f, v, C_max, R_max) with the redemption request for auditability.

DATA MODEL (Drizzle migrations)
1) profit_policy (single active row, versioned)
- id uuid pk
- effective_from timestamp
- min_margin_m numeric (default 0.25)
- affiliate_rate_a numeric (default 0.02)  // configurable per source later
- affiliate_haircut_h numeric (default 0.70)
- processing_fee_rate_r numeric (default 0.00)
- fixed_fee_f_cents int (default 0)
- packpts_value_v_microusds int  // e.g., $0.002 => 2000 microusd per PackPTS
- enabled boolean default true
- created_at timestamp

2) external_purchase_intent (tracks the listing and the redemption calculation)
- id uuid pk
- user_id uuid
- source enum('ebay','goldin')
- listing_id text
- listing_url text
- price_cents int
- currency text default 'usd'
- computed_rmax int
- requested_redeem_packpts int default 0
- approved_redeem_packpts int default 0
- status enum('CREATED','APPROVED','DENIED','PURCHASE_CONFIRMED','CREDIT_GRANTED','CANCELED')
- calc_snapshot jsonb (store P,A,h,m,r,f,v,Cmax,Rmax)
- created_at timestamp
- updated_at timestamp

3) redemption_credit (what we ultimately grant as store credit / rebate credit)
- id uuid pk
- purchase_intent_id uuid fk
- user_id uuid
- packpts_spent int
- credit_cents int
- status enum('PENDING','GRANTED','REVERSED')
- ledger_spend_entry_id uuid nullable
- ledger_credit_entry_id uuid nullable
- created_at timestamp

BACKEND ENDPOINTS
A) GET /api/profit-policy (public)
- returns active policy (m, v, etc.) for UI display, not secrets.

B) POST /api/marketplace/redemption/quote (auth required)
Body: { source, listingId, listingUrl, priceCents, currency }
- Load active profit_policy
- Compute:
  P = priceCents/100
  A,h,m,r,f,v
  Cmax and Rmax per formula
- Return:
  { rMax, creditCentsMax, policySummary, explanationText }
- Store an external_purchase_intent row with calc_snapshot and computed_rmax.

C) POST /api/marketplace/redemption/apply (auth required)
Body: { purchaseIntentId, requestedRedeemPackpts }
- Load purchase intent and active policy
- Recompute Rmax (do not trust stored value)
- approvedRedeemPackpts = min(requestedRedeemPackpts, Rmax, userWalletBalance)
- If approvedRedeemPackpts <= 0 => DENIED
- Else:
  - Spend PackPTS via ledger (type=SPEND) using idempotency key "redeem:<purchaseIntentId>"
  - Create redemption_credit with packpts_spent, credit_cents = floor(approvedRedeemPackpts * v)
  - Mark purchase intent APPROVED and store approved amount
Return approved amounts.

D) GET /out/:source/:listingId?intent=<purchaseIntentId>
- This is our tracked redirect to eBay/Goldin listing.
- Log click.
- If purchaseIntentId exists and is APPROVED, include a “custom id” in affiliate params if supported.
- Redirect to destination URL.

E) POST /api/marketplace/purchase/confirm (auth required)
Body: { purchaseIntentId, evidence }
- For now, implement MANUAL confirmation stub (admin review) OR “user uploads receipt” placeholder.
- Mark status PURCHASE_CONFIRMED.
- If confirmed, grant store credit internally (closed-loop) OR trigger a future automation.
- IMPORTANT: do NOT cash out. Credit is applied within PackPts store as store credit or next purchase discount.

ADMIN ENDPOINTS
