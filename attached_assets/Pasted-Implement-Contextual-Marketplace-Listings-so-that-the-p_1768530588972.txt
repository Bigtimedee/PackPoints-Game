Implement “Contextual Marketplace Listings” so that the physical card listings shown in PackPts Marketplace always correlate to the card types + vintages users are actively playing with in PackPTS matches (e.g., 1987 Topps Baseball, 1992 Upper Deck Basketball). This must work for eBay (official API) and Goldin (curated/partner feed stub), and integrate into existing gameplay + WebSocket match system.

STACK (do not change)
Frontend: React 18 + Vite + TS + Tailwind + shadcn/ui + TanStack React Query + Wouter
Backend: Express + TS + Passport + express-session + ws (1v1 realtime)
DB: Postgres (Neon) + Drizzle ORM + Zod
Payments: Stripe (existing)
Email: Nodemailer (existing)

GOALS
1) Define a canonical “Game Catalog” of playable card sets/vintages (e.g., 1987 Topps Baseball; 1992 Upper Deck Basketball).
2) Ensure Marketplace results are filtered to the same “active catalog contexts” a user is currently playing, by default.
3) Provide UX controls so the user can switch contexts (e.g., “1987 Topps Baseball” tab) or broaden search intentionally.
4) Keep listings current via eBay official API + TTL caching; for Goldin, use curated/partner feed filtered by context tags.
5) Create robust query generation and validation so we only fetch relevant vintage/product listings (avoid drift).

NON-NEGOTIABLE CONSTRAINTS
- Never scrape eBay/Goldin pages; use eBay official Browse search. Goldin is curated/partner feed until official feed exists.
- The “context” filter must be enforced server-side (don’t rely on client).
- All context selection and “what user is playing” signals must be stored server-side.
- Use TTL caching for external search results (default 5 minutes) keyed by (context + query + sort + limit).
- Provide a safe fallback if no results: broaden query slightly but stay within the same vintage + sport.

DATA MODEL (Drizzle migrations)
A) game_sets (canonical playable contexts)
- id uuid pk
- sport enum('baseball','basketball','football','hockey')
- brand text (e.g., 'Topps','Upper Deck')

- set_name text (e.g., 'Topps Baseball' or 'Upper Deck Basketball')
- league text nullable (MLB/NBA/NFL/NHL)
- is_active boolean default true
- marketplace_keywords jsonb (base keywords; e.g., ["1987 Topps","Topps 1987","1987 Topps Baseball"])
- created_at timestamp

B) user_game_preferences
- user_id uuid pk
- active_set_ids uuid[] (or a join table user_active_sets)

- updated_at timestamp

Preferred normalized join table:
- user_active_sets: id, user_id, game_set_id, last_used_at, is_default bool

C) match_context_log (connect gameplay -> context)
- id uuid pk
- user_id uuid
- match_id uuid nullable
- game_set_id uuid
- event_type enum('MATCH_STARTED','MATCH_COMPLETED','SET_SELECTED')
- created_at timestamp
Indexes: (user_id, created_at desc), (game_set_id)

D) external_search_cache (if not already)
- source enum('ebay','goldin')
- cache_key text unique
- payload jsonb
- expires_at timestamp

E) goldin_curated_listings (existing or create)
- id uuid
- title text

- tags text[]  // MUST include normalized context tags, e.g. ["sport:baseball","year:1987","brand:Topps"]
- is_active boolean

BACKEND LOGIC
1) Canonical context normalization
Create a “context key” builder:
contextKey = `${sport}:${year}:${brand}` (e.g., "baseball:1987:Topps")
Store in game_sets as computed/virtual or in code.

2) Integrate context with gameplay
Where matches are created/started (HTTP and/or WebSocket start event):
- Require a game_set_id for the match OR derive from the selected pack/mode.
- Record match_context_log with MATCH_STARTED and game_set_id for each participating user.
On match complete:
- Record MATCH_COMPLETED entries.
Also add endpoint:
- POST /api/game/active-sets
  Body: { gameSetIds: uuid[], defaultSetId?: uuid }
  Updates user_active_sets and records SET_SELECTED events.

3) Determine “active contexts” for a user (server-side)
Implement function getUserActiveContexts(userId):
- Return up to N game_set_ids based on:
  a) user_active_sets default + selected
  b) last N match_context_log entries (e.g., last 7 days), weighted by recency
- If none, return system defaults (e.g., most popular sets).

4) Marketplace search endpoint (context-aware by default)
Implement:
GET /api/marketplace/contextual-search
Query params:
- q (optional; user free-text)
- source ('ebay'|'goldin'|'all', default 'all')
- setId (optional; if provided, force that set)
- limit, sort
Behavior:
- Determine contexts:
  - If setId provided and user is allowed to view it, use that one
  - Else use getUserActiveContexts(userId) if logged in
  - If not logged in, use global default active contexts
- For each context, construct a marketplace query using ONLY that context:
  - baseKeywords = game_sets.marketplace_keywords
  - if q provided, append sanitized q but keep context keywords required
  - Examples:
    - "1987 Topps Baseball Kirby Puckett"
    - "1992 Upper Deck Basketball Shaquille O'Neal"
- Call eBay Browse search for each context (or only default context unless user expands).
- Return results grouped by context:
  { contexts: [{ gameSet, listings, lastUpdated }], appliedContextIds, noteIfBroadened }

IMPORTANT: enforce that every eBay query includes year + brand + sport tokens; do not allow a query that omits them when in contextual mode.

5) Query broadening fallback (within same vintage)
If contextual search returns < MIN_RESULTS (e.g., 5):
- Broaden in a controlled way:
  - Drop player name terms, keep year+brand+sport.
  - Optionally add grade/condition variants.
- Never broaden outside the year+brand+sport unless the user explicitly chooses “Show all years/brands.”

6) Goldin filtering (curated/partner)
For Goldin:
- Filter goldin_curated_listings by required tags derived from contextKey.
- Return those items as listings with source='goldin'.

7) Caching
Cache key must include:
- source + contextKey + (sanitized q) + sort + limit
TTL: 5 minutes.
Allow “forceRefresh=1” with rate limit (max 1 per minute per user/IP).

8) Frontend routes & UX
A) Marketplace page:
- On load (logged in):
  - call GET /api/marketplace/contexts to fetch active contexts + defaults
  - show tabs/chips for each context (e.g., “1987 Topps Baseball”, “1992 Upper Deck Basketball”)
  - default tab = user defaultSetId or most recent match context
- Search bar:
  - searches within selected context by default
  - toggle: “Search across my active sets” (multi-context)
  - toggle: “Show all years/brands” (explicit broad mode; show warning)
- Listings:
  - grouped by selected context
  - show “Matched to your current game set: 1987 Topps Baseball” badge

B) Gameplay -> Marketplace deep link
After match completion, show CTA:
- “Buy cards from this set” -> routes to /marketplace?setId=<gameSetId>

9) Admin tools
- CRUD game_sets (activate/deactivate sets, manage marketplace keywords)
- CRUD goldin curated listings (with required tags)

SECURITY / DATA QUALITY
- Validate all game_set_id inputs with Zod.
- Ensure user can only select from game_sets where is_active=true (unless admin).
- Ensure contextual marketplace queries always include required vintage tokens.

TESTS

- contextual-search enforces required tokens and cannot be bypassed by client
- caching works per contextKey
- fallback broadening stays within year+brand+sport
- UI loads default context from last match, and tab switching changes results
