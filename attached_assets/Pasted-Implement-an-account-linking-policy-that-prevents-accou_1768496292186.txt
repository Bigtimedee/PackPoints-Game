Implement an account-linking policy that prevents account takeover (ATO) when linking identities across providers (local username/password, Replit OAuth, WorkOS). This must be implementation-ready and integrated into the existing stack: Express + TypeScript + Passport + express-session + Postgres (Neon) + Drizzle, React/Vite frontend.

GOAL
Prevent “same email == same account” auto-linking. Email can be used for UX hints only. Linking must require an explicit proof of control of the existing PackPts account.

CORE SECURITY PRINCIPLES (NON-NEGOTIABLE)
1) Never auto-link accounts based solely on email, even if a provider claims the email is verified.
2) Linking requires one of:
   - user is already authenticated to the PackPts account in the current session and explicitly initiates linking, OR
   - user completes a link-challenge that proves control of the existing PackPts account (login OR magic-link).
3) A provider identity (provider+providerUserId) can link to only one PackPts user (unique constraint).
4) If a WorkOS/Replit identity is already linked to a different PackPts user, block linking and require support/manual merge.
5) For “high-value” accounts (PackPTS balance >= threshold OR Stripe subscriber OR redemption history), require stronger proof (login + 2FA if available OR magic-link), and never allow “email-only” linking.

E) OPTIONAL high-value signals
- redemptions table exists OR wallet balance exists OR stripe_customer/subscription exists.
Implement a function isHighValue(userId) that returns true if:
- wallet.balance >= HIGH_VALUE_PACKPTS_THRESHOLD (config), OR
- has active Stripe subscription, OR
- has any redemption history.

Implement linking logic identically for replit/workos. Local is the “anchor” account type.

2) LOGIN VIA PROVIDER WHEN USER IS NOT LOGGED IN
On provider callback (WorkOS or Replit):
Inputs: providerUserId, providerEmail, providerEmailVerified (if available)

Decision tree:

CASE A: Identity already linked
- If identity exists for (provider, providerUserId):
  -> log in that linked user (req.login)
  -> audit LINK_COMPLETED (reason: existing identity)
  -> DONE

CASE B: No identity, but provider email matches an existing PackPts user
- existingUser = first user whose primary_email == providerEmail OR any identity.email == providerEmail
- If existingUser found:
  -> DO NOT auto-link
  -> Create pending link challenge tied to the CURRENT session:
       - store challenge id in req.session.pendingLinkChallengeId
       - set expires_at = now + 15 minutes
  -> Redirect to FRONTEND /auth/link-required with context:
       - show masked email and provider name
  -> audit LINK_BLOCKED (reason: email collision; requires proof)
  -> DONE

CASE C: No identity and no email match
- Create new user with primary_email = providerEmail (if you enforce email uniqueness; otherwise store on identity only)
- Create identity linking provider->new user
- req.login(newUser)
- audit LINK_COMPLETED (reason: new user)
- DONE

3) LINK PROVIDER TO AN EXISTING LOGGED-IN USER (SAFE PATH)
When a user is already authenticated (req.user exists) and chooses “Link WorkOS” or “Link Replit”:
- Start provider flow with a special parameter in session: req.session.linkIntent = true
On provider callback:
- If identity exists and identity.userId != req.user.id:
   -> Block
   -> audit LINK_BLOCKED (reason: identity already linked to another user)
   -> redirect /auth/error?code=IDENTITY_IN_USE
- Else:
   -> Create identity linking provider->req.user.id
   -> audit LINK_COMPLETED (reason: user-initiated link while logged in)
   -> redirect /settings/accounts?linked=1

4) RESOLVING A PENDING LINK CHALLENGE (NO AUTO-LINK)
Frontend page /auth/link-required provides two proof options:
A) “Log in to my existing PackPts account” (local username/password OR existing Replit auth)
B) “Email me a verification link” (magic-link) — only if providerEmail equals existingUser.primary_email and that email is verified on existing account.

Backend endpoints:

POST /api/auth/link/resolve/login
- User logs in via existing login routes (no special changes) but AFTER successful login:
  - If req.session.pendingLinkChallengeId exists:
     -> call POST /api/auth/link/confirm (see below)
     -> redirect success

POST /api/auth/link/send-magic
- Requires session has pendingLinkChallengeId
- Looks up matching existing user by email
- If no user or email not verified => deny
- Send one-time token link to that email:
   /api/auth/link/verify?token=...
- Store token hashed in DB with expiry (15 min)

GET /api/auth/link/verify?token=...
- Validates token, logs user in (or prompts to login), then confirms linking by calling /api/auth/link/confirm

POST /api/auth/link/confirm
- Requires:
  - req.user authenticated (proof of control)
  - pendingLinkChallengeId exists and not expired
- Load challenge (provider, providerUserId, email)
- SECURITY CHECKS:
  - If identity already exists for (provider, providerUserId) and identity.userId != req.user.id => block
  - If isHighValue(req.user.id) => require additional step if you support 2FA; otherwise require magic-link confirmation (must have been completed)
- If passes:
  - Create identity linking provider->req.user.id
  - Mark challenge COMPLETED, clear session pending id
  - audit LINK_COMPLETED (reason: challenge resolved)
  - Return success payload

5) HIGH-VALUE ACCOUNT ESCALATION
If isHighValue(existingUserId) OR (wallet.balance >= threshold OR active subscription OR redemption history):
- Do not allow linking via “just logged in once” unless:
  - user completed 2FA (if your app has it), OR
  - user completed magic-link confirmation to primary_email within same session.
Implement configuration:
- HIGH_VALUE_PACKPTS_THRESHOLD (default 10000)
- HIGH_VALUE_REQUIRE_MAGIC_LINK (default true)
If required and not satisfied => redirect /auth/link-required?step=verify-email

6) EMAIL UNIQUENESS POLICY (RECOMMENDED)
Enforce users.primary_email unique.
- If providerEmail exists on another user:
  - You may not create a new user with same email
  - Must go through link-required flow
If you cannot enforce uniqueness now, then:
- Permit duplicates ONLY if the duplicate account is marked LIMITED (cannot redeem, cannot transfer, cannot withdraw, cannot exceed minimal PackPTS).
But strongly prefer uniqueness.

7) ACCOUNT MERGE POLICY (DO NOT AUTOMATE)
- If two accounts exist for the same person, do NOT auto-merge PackPTS wallets or purchase history.
- Provide an admin-only “merge request” record and manual support workflow.
- Add endpoint POST /api/support/merge-request that captures details and logs audit MERGE_REQUESTED.
