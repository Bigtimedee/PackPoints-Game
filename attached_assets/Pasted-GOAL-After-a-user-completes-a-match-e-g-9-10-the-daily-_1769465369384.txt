GOAL: After a user completes a match (e.g., 9/10), the “daily counter” in the top-left (0/200 cards) must immediately update to reflect the user’s daily progress (cards completed today), without requiring a refresh. Implement server-authoritative daily progress tracking + client invalidation/update on match completion.

STACK
React 18 + Vite + TS + TanStack Query
Express + TS
Postgres (Neon) + Drizzle

DEFINITION
- Daily “cards completed” = total questions (cards) the user has answered today across ALL match modes.
- Daily cap target = 200 cards/day (10 matches * 20 cards OR any combination; you already track 0/200).
- The counter should update:
  1) immediately after MATCH_END (or after final submit/advance)
  2) also on page focus/reload
- Use the user’s local day (or server day) consistently. Choose SERVER day in America/Chicago unless you already have a user timezone field.

========================================================
A) DATABASE

Create a durable per-day rollup table:

Table: user_daily_progress
- user_id uuid not null
- day_date date not null               -- server date, e.g. CURRENT_DATE in America/Chicago
- cards_answered int not null default 0
- matches_completed int not null default 0
- updated_at timestamp not null default now
Primary key (user_id, day_date)

(OPTIONAL) also store pts_earned_today, etc. if needed.

========================================================
B) SERVER: WRITE DAILY PROGRESS ON MATCH COMPLETION (AND/OR EACH ANSWER)

Wherever your match engine finalizes a match (FINISHED end_reason='completed'):
- For each participant, compute how many answers they submitted for this match (should equal questions_total)
- Increment their cards_answered by that amount
- Increment matches_completed by 1
Do this inside the same DB transaction that finishes the match, so it’s consistent.

Implementation steps:

1) Create helper:
src/services/progress/dailyProgress.ts

export async function bumpDailyProgressForMatch({
  tx, userId, cardsDelta, matchesDelta
}) {
  // UPSERT user_daily_progress for (userId, day_date)
  // day_date must be computed in server timezone (America/Chicago)
}

2) Determine day_date in America/Chicago:
- Use a single function getChicagoDate(): date-only string
- Don’t rely on JS Date() default timezone.
(If you already have timezone infra, use that.)

3) After match finishes (and ONLY once per match):
- Ensure idempotency: don’t double count if finish is retried.
Add a boolean or marker:
- matches.progress_applied boolean default false
When finishing:
- if progress_applied == false:
    apply bumps for host and guest
    set progress_applied = true

========================================================
C) SERVER: API TO READ DAILY PROGRESS

Add endpoint:
GET /api/progress/daily
Auth required.
Returns:
{
  dayDate: "2026-01-26",
  cardsAnswered: 120,
  matchesCompleted: 6,
  capCards: 200
}

Implementation:
- SELECT from user_daily_progress where user_id=req.user.id and day_date = todayDate
- If none, return zeros.

========================================================
D) CLIENT: TANSTACK QUERY SOURCE OF TRUTH

1) Create query:
useDailyProgress()
- queryKey: ["dailyProgress"]
- queryFn: fetch("/api/progress/daily", { credentials:"include" }).json()

2) Header component (top-left counter) must render from this query:
cardsAnswered / capCards
Example: `${cardsAnswered}/${capCards} cards`

3) On match completion, force refresh:
- When receiving WS event MATCH_END OR after final submit resolves:
    queryClient.invalidateQueries({ queryKey: ["dailyProgress"] })

If you already have WS in the match page:
- In the MATCH_END handler, call invalidateQueries.
- Also in PLAY AGAIN / BACK TO HOME click handlers, invalidateQueries.

4) Optimistic UI (optional but recommended):
- Immediately after MATCH_END, do:
  queryClient.setQueryData(["dailyProgress"], (prev) => ({
    ...prev,
    cardsAnswered: prev.cardsAnswered + questionsTotalThisMatch,
    matchesCompleted: prev.matchesCompleted + 1
  }))
Then still invalidate to reconcile with server.

========================================================
E) ENSURE DAILY COUNTER UPDATES WITHOUT NAVIGATION

If the match-end screen is still showing the header:
- Ensure the header component remains mounted and uses the same queryClient instance (single QueryClientProvider at app root).
- If you have multiple QueryClientProviders, remove duplicates and keep one at top-level.

========================================================
F) ACCEPTANCE CRITERIA

- Finish a match -> top-left counter immediately increments by the number of cards in the match (e.g., +10 or +20 depending on your match format).
- No full refresh required.
- Counter remains correct across reloads.
- Progress is not double-counted if MATCH_END is received twice or engine retries (progress_applied safeguard).
