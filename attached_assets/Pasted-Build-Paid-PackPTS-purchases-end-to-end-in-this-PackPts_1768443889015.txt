Build “Paid PackPTS” purchases end-to-end in this PackPts stack (React 18 + Vite + TS + Tailwind + shadcn/ui + React Query + Wouter; Express + TS; Postgres (Neon) + Drizzle; Passport + express-session; Stripe).

GOAL
Allow users to buy PackPTS bundles (e.g., 6,000 PackPTS for $9.99) via Stripe Checkout, and credit PackPTS ONLY after verified payment. Must be secure, idempotent, and ledger-based.

NON-NEGOTIABLE CONSTRAINTS
- PackPTS is credited server-side only, via an append-only ledger (no client-side points mutation).
- Stripe payments must be verified via webhook; the client never “tells” the server it paid.
- Every credit operation must be idempotent (unique on Stripe event id and/or checkout session id).
- Support refunds/chargebacks by reversing PackPTS credits (REVERSAL ledger entries) where applicable.
- Use existing auth/session middleware; only logged-in users can purchase.

BACKEND IMPLEMENTATION (Express + Drizzle)
1) Database schema (Drizzle migrations)
Create/extend these tables (adapt to existing naming if already present):
- products:
  - id, sku UNIQUE, name, type enum('PACKPTS_BUNDLE'), packpts_grant int, price_cents int, currency text, is_active bool, created_at
Seed at least:
  - SKU: PACKPTS_6000, name “6,000 PackPTS”, packpts_grant=6000, price_cents=999, currency='usd'
  - plus 2 more bundles (e.g., 1500 for 2.99, 15000 for 19.99)

- stripe_checkout_sessions:
  - id, user_id, sku, stripe_session_id UNIQUE, status enum('CREATED','PAID','CANCELED','EXPIRED'), created_at, updated_at, metadata json

- purchase_events:
  - id, provider enum('stripe'), event_id UNIQUE, event_type, payload json, processed_at, status enum('RECEIVED','PROCESSED','FAILED'), error text

- wallet + ledger_entry:
  - If not already present, implement wallet/ledger as append-only:
    - ledger_entry includes entry_type enum('EARN','SPEND','ADJUST','PURCHASE_CREDIT','REVERSAL'), amount int, reason text, metadata json, idempotency_key UNIQUE, created_at
    - wallet balance is updated only inside DB transactions when inserting ledger entries.

2) Stripe integration
- Add Stripe server SDK and env vars:
  - STRIPE_SECRET_KEY=
  - STRIPE_WEBHOOK_SECRET=
  - STRIPE_SUCCESS_URL= (e.g., APP_BASE_URL + '/store/success?session_id={CHECKOUT_SESSION_ID}')
  - STRIPE_CANCEL_URL= (e.g., APP_BASE_URL + '/store/cancel')
- Create endpoint:
  POST /api/store/checkout
  Body: { sku: string }
  Auth required.
  Behavior:
   - Validate sku is active and is PACKPTS_BUNDLE
   - Create Stripe Checkout Session (mode='payment') with line item price_data from DB
   - Attach metadata: userId, sku, packptsGrant
   - Save stripe_session_id in stripe_checkout_sessions with status CREATED
   - Return { url } for client redirect

3) Webhooks (the source of truth)
- Implement POST /api/webhooks/stripe with raw body parsing for signature verification.
- Handle at minimum:
  a) checkout.session.completed:
     - Verify session.payment_status == 'paid'
     - Read metadata userId + sku + packptsGrant
     - In a DB transaction:
        - mark stripe_checkout_sessions as PAID
        - insert ledger_entry type=PURCHASE_CREDIT, amount=packptsGrant,
          reason='Stripe PackPTS Purchase',
          idempotency_key = 'stripe:' + session.id (or event.id) UNIQUE
        - update wallet balance accordingly
     - Record purchase_events as PROCESSED
  b) charge.refunded or refund events:
     - Locate original session/event
     - Insert ledger_entry type=REVERSAL for the same amount (or proportional) with idempotency_key unique
     - Mark stripe_checkout_sessions status appropriately
  c) payment_intent.payment_failed (optional): mark FAILED/EXPIRED

- Ensure webhook is fully idempotent: if Stripe retries the event, we do not double-credit.

4) Read endpoints for UI
- GET /api/store/products -> returns active PackPTS bundles
- GET /api/wallet -> returns current PackPTS balance + recent ledger entries
- GET /api/store/checkout/:sessionId -> returns status (CREATED/PAID/etc.) so success page can confirm.

FRONTEND IMPLEMENTATION (React + Vite + shadcn + React Query + Wouter)
1) Create a Store page route /store
- Fetch /api/store/products
- Render product cards (shadcn Card + Button) with “Buy” CTA
- On Buy: POST /api/store/checkout {sku} then redirect browser to returned Stripe Checkout URL

2) Success/Cancel routes
- /store/success?session_id=...
  - Poll GET /api/store/checkout/:sessionId until status=PAID (with timeout)
  - Then refetch /api/wallet and show “PackPTS added!” + new balance
- /store/cancel
  - Show friendly message and link back to store
