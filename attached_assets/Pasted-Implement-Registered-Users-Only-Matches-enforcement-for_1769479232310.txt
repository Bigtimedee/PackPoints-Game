Implement “Registered-Users-Only Matches” enforcement for BOTH 1vFriend and 1vRandom. A match must ONLY be created/joined/played if BOTH participants are authenticated PackPTS users (have a valid userId in session) and are validated against the DB. This must be enforced server-side (not just UI).

STACK: React + TS, Express + TS, ws, Postgres + Drizzle, Passport + express-session.

========================================================
A) DEFINE WHAT “REGISTERED USER” MEANS
- Server session contains req.user.id (Passport) OR ws.userId derived from the same session/cookie.
- User exists in DB users table and is not disabled/banned.
- (Optional) verified_email == true if you require it.

Create helper:
src/auth/requireAuth.ts
- For REST: middleware requireAuth(req,res,next) -> 401 if !req.user
- For WS: attach userId to ws during upgrade; if absent, close connection.

Create helper:
src/auth/requireActiveUser.ts
- Loads user row; rejects if disabled/banned.

========================================================
B) 1vRANDOM: ONLY AUTHENTICATED USERS CAN ENTER QUEUE
1) WS event: QUEUE_JOIN { mode:"1vRandom", setId, questionsTotal }
Server rules:
- If !ws.userId -> send ERROR {code:"NOT_AUTHENTICATED"} and DO NOT queue.
- Confirm user exists and is active.
- Enforce “one active queue entry per user” (unique by userId).
- Store queue row in DB (not memory) for reliability:

Table: match_queue
- user_id uuid PK
- mode text
- set_id uuid
- questions_total int
- joined_at timestamp
- last_seen_at timestamp

2) Server matchmaking loop (or on each QUEUE_JOIN):
- Take two distinct user_ids from match_queue (oldest first)
- Create match row with host_user_id and guest_user_id (both from DB users)
- Remove both from queue in same transaction
- Broadcast MATCH_FOUND to each user’s sockets

3) WS event: QUEUE_LEAVE clears match_queue row.

========================================================
C) 1vFRIEND: ONLY REGISTERED USERS CAN INVITE + JOIN

Preferred approach: invite by username/userId (NOT open public join code).
Implement:
POST /api/matches/friend/create
Body: { opponentUserId, setId, questionsTotal }
Server rules:
- requireAuth + requireActiveUser
- opponentUserId must exist and be active
- opponentUserId != current user
- create match with:
  host_user_id = current user
  guest_user_id = opponentUserId
  status='LOBBY'
  invite_token (random) OPTIONAL for UX, but NOT the sole gate

Joining:
POST /api/matches/:matchId/friend/join
Server rules:
- requireAuth
- match must be status='LOBBY'
- req.user.id MUST equal guest_user_id (or host_user_id rejoin)
- otherwise 403

If you still use “join code” UI:
- Join code only resolves to a matchId, but server still checks:
  req.user.id == guest_user_id (or host)
- If not, reject (403) and do NOT reveal match data.

========================================================
D) MATCH ENGINE: HARD AUTHORIZATION ON EVERY ACTION

For ALL WS events (ANSWER_SUBMIT, JOIN_MATCH, MATCH_RESYNC, etc):
- Load match participants (host_user_id, guest_user_id).
- If ws.userId is not one of them -> 403 NOT_AUTHORIZED.
This ensures non-registered users cannot play even if they guess a matchId.

Add helper:
src/services/matches/assertParticipant(matchId, userId)

========================================================
E) CLIENT UI: PREVENT ENTRY WHEN NOT LOGGED IN (SECONDARY)
- For Play -> 1vRandom or 1vFriend buttons:
  - If currentUser == null, route to /login and show “Sign in required”.
- But do not rely on this—server rules above are the real enforcement.

========================================================
F) ACCEPTANCE RULES
- A logged-out visitor cannot join queue, create/join friend match, or submit answers (401/WS close).
- A registered user can only start a match against another registered user that exists in DB.
- Any attempt to join a friend lobby by someone other than the intended guest returns 403.
- Any attempt to send WS gameplay events by non-participants returns 403 and is ignored.

DELIVERABLES
- requireAuth middleware (REST) + ws user attach/close if unauthenticated
- DB-backed match_queue + matchmaking transaction
- friend match create/join endpoints that validate opponent user exists + active
- assertParticipant enforced on every match WS handler
