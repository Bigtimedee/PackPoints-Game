Build the PackPTS Fraud Scoring Pipeline end-to-end (DB tables + rollups + signal computation + risk snapshots + minimal APIs) so the Payment Risk Engine can consume a simple, explainable risk snapshot per user.

STACK (DO NOT CHANGE)
Frontend: React 18 + Vite + TypeScript + Tailwind + shadcn/ui + TanStack React Query + Wouter
Backend: Express + TypeScript (tsx dev)
Realtime: ws
DB: Postgres (Neon) + Drizzle ORM + migrations
Auth: Passport + express-session
Payments: Stripe

OBJECTIVES
1) Create append-only event tables for: auth, device, payments, redemption, gameplay.
2) Create rollup tables for user/device/ip activity (10m/1h/24h windows; implement 24h first).
3) Compute concrete, explainable fraud signals from rollups + recent events.
4) Store signals append-only and maintain a single “risk_snapshot” row per user.
5) Provide minimal internal APIs so the Payment Risk Engine can read:
   - risk tier suggestion (LOW/MEDIUM/HIGH)
   - flags and top reasons
   - key recency fields (last purchase, last redemption apply, last device, last country)
6) Ensure jobs are safe, idempotent, retryable, and lightweight enough for Replit.

SECURITY
- Do not store raw IP; store ip_hash only (SHA-256 with server salt).
- device_id is a hashed stable fingerprint (privacy safe).
- All admin/internal endpoints must require admin role (or internal token).
- Do NOT expose full risk evidence to non-admin users.

SECTION D — ROLLUP COMPUTATION (24H MVP)
Implement:
- src/services/risk/rollups24h.ts
  - computeUserRollup24h(userId, windowStartDateUTC)
  - computeDeviceRollup24h(deviceId, windowStartDateUTC)
  - computeIpRollup24h(ipHash, windowStartDateUTC)

Rollup logic (use SQL aggregates):
User rollup:
- login_fail_count: COUNT auth_events LOGIN_FAIL in 24h
- login_success_count: COUNT auth_events LOGIN_SUCCESS in 24h
- distinct_device_count: COUNT DISTINCT device_id across auth_events/device_events/payment_events/redemption_events in 24h
- distinct_ip_count: COUNT DISTINCT ip_hash across auth_events/device_events/payment_events in 24h
- purchase_count / amount: COUNT + SUM(payment_events where event_type='PAID' in 24h)
- redemption_apply_count / pts_approved: COUNT redemption_events APPLY and SUM pts_approved in 24h
- gameplay_matches: COUNT DISTINCT match_id from gameplay_events MATCH_END
- gameplay_answers: COUNT gameplay_events ANSWER_SUBMITTED
- gameplay_correct: COUNT where answer_correct=true
- gameplay_median_response_ms: compute median (approx ok: percentile_cont(0.5) if available; else pull values and compute in TS)

Device rollup:
- distinct_user_count: COUNT DISTINCT user_id from device_events/payment_events/auth_events for that device in 24h
- purchase_count/amount: from payment_events for that device in 24h
- signup_count: auth_events SIGNUP for that device in 24h

IP rollup:
- same as device rollup but grouped by ip_hash

Upsert rollup tables each run.

SECTION E — SIGNAL COMPUTATION (EXPLAINABLE)
Implement:
- src/services/risk/signals.ts
  - computeSignalsForUser24h(userId): returns array of {signal_type, severity, evidence}
  - respects risk_suppressions (ignore active suppressions)
  - writes risk_signals append-only (idempotent per day: use deterministic hash key in evidence and skip duplicates)

Signals to compute (MVP list; EXACT):
1) NEW_DEVICE (severity 1)
- Trigger: user’s last_device_id differs from current device_id in last 24h AND device not seen in last 90d (approx: not seen in last 30d for MVP).
Evidence: {current_device_id, prior_device_id, first_seen_at}

2) DEVICE_CHURN (severity 1..3)
- Trigger: user_rollup.distinct_device_count >= 2
- severity: 1 if 2 devices, 2 if 3 devices, 3 if >=4
Evidence: {distinct_device_count}

3) NEW_COUNTRY (severity 2)
- Trigger: current ip_country != risk_snapshot.last_country (if both non-null) within 24h
Evidence: {last_country, current_country}

4) IP_VELOCITY (severity 2..3)
- Trigger: ip_rollup.distinct_user_count >= 3
- severity: 2 if 3-4, 3 if >=5
Evidence: {ip_hash, distinct_user_count}

5) DEVICE_MULTI_ACCOUNT (severity 3)
- Trigger: device_rollup.distinct_user_count >= 3
Evidence: {device_id, distinct_user_count}

6) PURCHASE_VELOCITY_MED (severity 1)
- Trigger: user_rollup.purchase_count == 2 OR purchase_amount_cents in [1500,3000)
Evidence: {purchase_count, purchase_amount_cents}

7) PURCHASE_VELOCITY_HIGH (severity 3)
- Trigger: user_rollup.purchase_count >= 3 OR purchase_amount_cents >= 3000
Evidence: {purchase_count, purchase_amount_cents}

8) REDEEM_SOON_AFTER_BUY (severity 3)
- Trigger: redemption APPLY within 10 minutes after a payment PAID event
Evidence: {paid_at, redeem_apply_at, minutes_diff}

9) EXTERNAL_REDEEM_PRESSURE (severity 2)
- Trigger: redemption_apply_count >= 3 OR redemption_pts_approved >= 3000 in 24h
Evidence: {redemption_apply_count, redemption_pts_approved}

10) BOT_FAST_RESPONSES (severity 1..3)
- Trigger: gameplay_answers >= 20 and gameplay_median_response_ms < 350
- severity: 1 if <350, 2 if <300, 3 if <250
Evidence: {gameplay_answers, gameplay_median_response_ms}

11) LOGIN_FAIL_SPIKE (severity 2..3)
- Trigger: login_fail_count >= 10 (2) or >=20 (3) in 24h (use 24h for MVP; later add 1h)
Evidence: {login_fail_count}

12) DISPUTE_PRESENT (severity 3)
- Trigger: any payment_events DISPUTE_OPENED in last 90d (for MVP use last 30d)
Evidence: {most_recent_dispute_at}

13) REFUND_SPIKE (severity 2)
- Trigger: >=2 REFUNDED events in last 30d
Evidence: {refund_count_30d}

NOTE: If ip_country missing, NEW_COUNTRY never triggers.

Process:

4) Write risk_signals entries.
5) Compute score (sum weights) using EXACT weights:
   DISPUTE_PRESENT +80
   DEVICE_MULTI_ACCOUNT +50
   PURCHASE_VELOCITY_HIGH +30
   REDEEM_SOON_AFTER_BUY +30
   IP_VELOCITY +25
   NEW_COUNTRY +20
   DEVICE_CHURN +20
   NEW_DEVICE +10
   BOT_FAST_RESPONSES +20
   LOGIN_FAIL_SPIKE +15
   PURCHASE_VELOCITY_MED +15

   REFUND_SPIKE +20

6) Determine tier_suggestion:

7) Build flags json:

8) Build top_reasons (max 3):

  - "Dispute present in last 30d"

9) Update risk_snapshot (upsert) with:

SECTION G — JOB RUNNER (DB QUEUE)
Implement lightweight worker loop:
- src/services/jobs/worker.ts

Behavior:
- poll jobs table every 2-5 seconds
- pick next PENDING job with run_after <= now, set RUNNING atomically
- attempts++ ; on success set SUCCEEDED; on error set FAILED and store error; retry with backoff:
  run_after = now + min(2^attempts * 30s, 10m)

Job types:
- ROLLUP_24H: compute rollups for user/device/ip based on job payload
- COMPUTE_SIGNALS: computeSignalsForUser24h(userId)
- UPDATE_SNAPSHOT: updateRiskSnapshot(userId)

Create helper:
- enqueueRiskRecalc(userId, deviceId?, ipHash?) that enqueues:
  - ROLLUP_24H (user)
  - UPDATE_SNAPSHOT (user)  (snapshot calls signals internally)

Call enqueueRiskRecalc from:
- Stripe webhook payment events (PAID/REFUNDED/DISPUTE_OPENED)
- Redemption APPLY
- Auth LOGIN_FAIL spike (optional)
- Device seen event (once per session)
