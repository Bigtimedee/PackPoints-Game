Design and implement PackPTS expiration + liability controls for PackPts, in this existing stack:
React 18 + Vite + TS + Tailwind + shadcn/ui + React Query + Wouter
Express + TS, Passport + express-session
Postgres (Neon) + Drizzle ORM + migrations
Stripe payments/subscriptions
PackPTS uses an append-only ledger (required)

GOALS
1) Introduce configurable PackPTS expiration rules that reduce long-term liability without harming core gameplay.
2) Provide accounting-grade liability reporting: outstanding points, aged buckets, projected expirations, breakage.
3) Enforce expiration safely using server-side jobs and idempotent ledger entries (no client trust).
4) Ensure legal/product constraints: PackPTS is promotional rewards, NOT cash; expiration must be transparent in UI.

NON-NEGOTIABLE CONSTRAINTS
- Append-only ledger: never delete or edit ledger entries.
- Expiration is implemented as additional ledger entries (type=EXPIRE) that reduce balance.
- Must be idempotent: no double-expire. Use unique idempotency keys for each expiry event.
- Expiration must not reduce a user’s balance below zero.
- Must support multiple “buckets” of points with different expiry dates (FIFO spending).
- Must support “exempt” points (e.g., purchased points or promo points) via policy flags.
- All logic server-side; UI only displays results.

PRODUCT POLICY (implement as configuration, not hard-coded)
A) EARNED points (from gameplay, streaks, referrals, promos):
- Expire 365 days after earned date by default.
- Inactivity rule: if no eligible gameplay activity for 90 days, expire all EARNED points older than 90 days (optional toggle).
B) PURCHASED points (from Stripe PackPTS bundles):
- Default: non-expiring OR 730-day expiry (configurable). Implement as separate policy.
C) BONUS points (one-time promos):
- Expire 30–180 days (configurable per promo).

D) User-visible rules:
- Always show upcoming expirations (next 30/60/90 days).
- Send reminder at 30 days and 7 days before expiration (email hooks only; do not build full campaigns).

Indexes: (user_id, expires_at), (user_id, status)

3) Create an expiration policy table:

BUSINESS LOGIC
1) When awarding PackPTS (EARN or PURCHASE_CREDIT):
- Continue writing a ledger_entry.
- ALSO create a packpts_bucket row:
  - remaining_amount = amount
  - expires_at computed from policy based on source_type (unless never-expiring)
  - For BONUS, allow override expiry days passed in metadata (validated server-side)
- IMPORTANT: buckets are the spend/expire unit; wallet balance is derived/updated by ledger transactions as you already do.

2) Spending PackPTS must be FIFO by expiry (earliest expires first), with source-type ordering:
- Spend buckets with earliest expires_at first; non-expiring buckets last.
- Create a spend_allocation table to record which buckets were reduced:
  - packpts_spend_allocation:
    - id, spend_ledger_entry_id, bucket_id, amount, created_at
- On spend:
  - insert SPEND ledger entry (amount negative or positive with type=SPEND; follow existing convention)
  - allocate decrements across buckets in a transaction; update remaining_amount/status
  - idempotency for spends remains required.

3) Expiration engine
- Create a server-side scheduled job (cron-like) that runs daily at 03:15 America/Chicago.
- It finds all OPEN buckets whose expires_at < now() - grace_period and remaining_amount > 0.
- For each user, expire eligible buckets in a transaction:
  - total_expire = sum(remaining_amount of expired buckets)
  - Insert EXPIRE ledger entry with amount = total_expire, reason='PackPTS Expiration', metadata includes bucket ids, and idempotency_key = 'expire:' + userId + ':' + asOfDate
  - Set buckets to EXPIRED and remaining_amount=0
- Idempotency must guarantee that reruns do not double-expire.

4) Inactivity expiration (optional toggle)
- If inactivity_enabled:
  - Determine user’s last eligible gameplay activity date (use match_completed timestamps).
  - If last activity older than inactivity_days:
    - Expire EARNED buckets where earned_at < now - inactivity_min_age_days
  - Same EXPIRE ledger entry mechanism and idempotency key: 'inactivity-expire:' + userId + ':' + asOfDate

API ENDPOINTS
User-facing:
- GET /api/wallet
  - include: current balance, breakdown by source_type, next_expiration_amount, next_expiration_date, and list of expirations within 90 days.
- GET /api/wallet/expirations?days=90
  - returns upcoming expiring buckets aggregated by week.

Admin-only:
- GET /api/admin/liability/summary?asOf=YYYY-MM-DD
  - totals + aged buckets + expiring windows.
- POST /api/admin/expiration/policy
  - create new effective policy (do not edit old ones)
- POST /api/admin/expiration/run
  - manual run for a given date range (protected)
- GET /api/admin/users/:id/buckets
  - view user buckets and expiry schedule.
