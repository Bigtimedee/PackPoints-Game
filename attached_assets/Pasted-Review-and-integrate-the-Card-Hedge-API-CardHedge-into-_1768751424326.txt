CARD HEDGE API REQUIREMENTS (from spec)
- Base URL: https://api.cardhedger.com
- Auth header: X-API-Key: <api-key> :contentReference[oaicite:1]{index=1}
Key endpoints we will use:
1) POST /v1/cards/card-search (flexible search, paging, raw_images_only option) :contentReference[oaicite:2]{index=2}
2) POST /v1/cards/search-cards-wsort (requires at least one filter; has sorting + paging) :contentReference[oaicite:3]{index=3}
3) POST /v1/cards/card-details (fetch card by card_id) :contentReference[oaicite:4]{index=4}
(Optionally later: /v1/cards/top-movers for “Trending” widgets, but not required for “complete sets”) :contentReference[oaicite:5]{index=5}

GOALS
1) Securely store CardHedge API key server-side and never expose to client.
2) Implement a backend CardHedge client with typed requests/responses + robust error handling.
3) Add a “Set Import” workflow that can pull all cards for a given set (ex: 1987 Topps Baseball) by paging through results and storing them locally in our DB so gameplay is fast and not dependent on CardHedge uptime.
4) Provide API endpoints for:
   - searching CardHedge directly (for admin tooling / debugging)
   - reading from our local “playable set” cache for gameplay
5) Provide a safe image strategy:
   - store and return card image URLs
   - add an optional image proxy endpoint to avoid mixed content / hotlink/CORS issues
6) Make it easy to seed canonical playable sets (Topps 1987 Baseball, Upper Deck 1992 Basketball, etc.) and map them into the existing “game_sets” / “contextual marketplace” system.

DATABASE (Drizzle migrations)
A) playable_sets
- id uuid pk
- sport enum('baseball','basketball','football','hockey')
- brand text   (Topps, Upper Deck, Panini, etc.)
- year int
- set_name text (exact as CardHedge “set” filter expects, e.g., "1987 Topps Baseball" OR whatever the API uses)
- cardhedge_set_query text (the exact query we will send to CardHedge; may equal set_name)
- category text (CardHedge category: "Baseball", "Basketball", etc.)
- is_active boolean default true
- created_at timestamp

B) playable_cards
- id uuid pk
- playable_set_id uuid fk playable_sets.id
- cardhedge_card_id text UNIQUE (CardHedge card_id)
- description text
- player text nullable
- set text
- number text nullable
- variant text nullable
- image_url text nullable
- category text nullable
- rookie boolean nullable
- raw_images_only boolean default false
- created_at timestamp
- updated_at timestamp
Indexes: (playable_set_id), (player), (set), (number)

C) cardhedge_import_runs
- id uuid pk
- playable_set_id uuid fk
- status enum('PENDING','RUNNING','SUCCESS','FAILED')
- started_at timestamp
- finished_at timestamp nullable
- page_size int
- pages_fetched int default 0
- cards_imported int default 0
- error text nullable

D) cardhedge_http_cache (optional)
- key text pk
- payload jsonb
- expires_at timestamp

BACKEND IMPLEMENTATION (Express + TS)
1) CardHedge HTTP client
Create src/services/cardhedge/client.ts:
- function cardHedgeFetch(path, method, body?) that:
  - adds headers: X-API-Key + Content-Type: application/json
  - handles timeouts, retries (max 2), and normalizes errors
- typed wrappers:
  - cardSearch(req): POST /v1/cards/card-search
  - cardSearchSorted(req): POST /v1/cards/search-cards-wsort
  - cardDetails(req): POST /v1/cards/card-details

2) Admin-only endpoints (protected)
A) POST /api/admin/cardhedge/search
- body: { search?, set?, category?, player?, rookie?, raw_images_only?, page, page_size }
- calls /v1/cards/card-search and returns response directly

B) POST /api/admin/cardhedge/search-sorted
- body: { search?, set?, category?, player?, raw_images_only?, page, page_size, sort_by, sort_order }
- enforce: at least one of (search, set, category, player) per spec :contentReference[oaicite:6]{index=6}
- calls /v1/cards/search-cards-wsort

C) POST /api/admin/playable-sets
- create a playable set record with: year/brand/sport/category + cardhedge_set_query

D) POST /api/admin/playable-sets/:id/import
- Kicks off an import run (synchronous is fine initially; if long, implement a background worker using a simple queue table)
- Implementation details:
  - Use pagination: start page=1, page_size=ENV default (<=100)
  - Use CardHedge search endpoint with EXACT set filter (req.set = playable_set.cardhedge_set_query) and category filter
  - Continue until no cards returned OR page exceeds “pages” returned in response
  - Upsert playable_cards by cardhedge_card_id
  - Store image URL from API (field named `image` in examples) :contentReference[oaicite:7]{index=7}
  - Record progress in cardhedge_import_runs

E) GET /api/playable-sets (public)
- returns active sets + counts (number of playable_cards per set)

F) GET /api/playable-sets/:id/cards
- returns cards from our DB (NOT from CardHedge) with paging + search by player/number/variant
- this will power gameplay packs

G) GET /api/cards/:cardhedgeCardId
- fetch from local DB; if not found, optionally fallback to CardHedge card-details and store result

3) Image handling (CORS-safe)
- Prefer: return image_url directly and let browser load it.
- Add optional proxy:
  - GET /api/images/proxy?url=<encoded>
  - Only allow whitelisted domains (from observed CardHedge image hosts)
  - Fetch server-side and stream with correct content-type, set cache headers
  - Prevent SSRF: reject private IPs, non-https, very long URLs
