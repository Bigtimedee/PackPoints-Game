PHASE 1 — PAYMENTS & MONEY SAFETY (HIGHEST RISK)
Goal: Zero chance of charging money without delivering value; zero sandbox leakage; full ledger integrity.

1) Stripe mode enforcement (LIVE on production host only):
- Ensure server-side Stripe key selection is host-locked (packpts.com => LIVE only).
- Remove any fallback to test keys; fail closed if live keys missing.
- Ensure frontend uses publishable key from /api/stripe/config, not embedded secrets.

2) Checkout creation:
- Validate bundle SKU/price mapping server-side only (never trust client).
- Validate amount, currency, and metadata (userId, bundleId).
- Success/cancel URLs must be correct per environment.
- Ensure idempotency keys used for session creation (prevent double charges).

3) Webhooks:
- Raw body signature verification.
- Live webhook secret on prod host.
- Idempotent processing of payment_intent.succeeded / checkout.session.completed.
- If webhook fails, queue retry (DB table “webhook_events” with status, attempts, last_error).
- Never credit PackPTS unless payment is confirmed.

4) PackPTS wallet/ledger:
- Implement a ledger table (if not present) with immutable entries:
  - id, user_id, type (credit/debit), source (purchase/gameplay/redemption/admin), ref_id, amount_packpts, created_at
- Balance is derived (sum credits - sum debits), NOT stored as a single mutable number (or store cached balance with reconciliation job).
- Add reconciliation job + admin report for mismatches.

5) Guardrails:
- Enforce min gross margin / max value per PackPTS in Admin package tool.
- Validate admin overrides logged with who/when/why.
- Block creation of bundles that violate guardrails unless explicit “override” with audit trail.

6) Fraud/abuse:
- Rate limit checkout creation per user/IP.
- Block purchases from unverified/unregistered users (must be registered for bundles crediting PackPTS).
- Add basic velocity checks.

Deliverable: /docs/PAYMENTS_AUDIT.md with “pass/fail” per item + fixes applied.

PHASE 2 — GAMEPLAY RELIABILITY (THE CORE PROMISE)
Goal: A user must always be able to complete a match as advertised. No missing submit button, no “match ended” at 1/10, no “waiting for opponent” deadlocks, no name leaks, no silhouettes/placeholder images.

1) Card serving contract:
- Define what “playable card” means:
  - has image URL that is NOT placeholder/silhouette/stock
  - image loads within timeout
  - player name can be reliably masked (top/bottom) OR card is excluded
- Add server-side “playable filter” gate BEFORE a card enters a set or match.

2) CardHedge integration hardening:
- Use CardHedge endpoints correctly:
  - card-search / search-cards-wsort with raw_images_only where available
  - pagination import correctness (page/page_size)
- Store CardHedge card_id as the canonical id.
- Ensure imports do NOT overwrite counts incorrectly (no accidental “purge all then fail insert”).
- Add transactional import: purge+insert in a DB transaction or staged table + swap.
- Add retry/backoff and logging for every API call (status code, latency).

3) “Silhouette purge” regression fix:
- Add deterministic detection rules:
  - Reject if image matches known placeholder domains/paths
  - Reject if image’s perceptual hash is near known silhouette hashes (store a small banned phash list)
  - Reject if CardHedge flags stock/placeholder (use raw_images_only).
- Ensure the UI never renders silhouettes as gameplay images; instead, request a new card server-side.

4) “Unable to load card image” behavior:
- Replace “Skip to next” that advances index with “Try different card” which:
  - requests a replacement card for the SAME slot number (i.e., card 1 remains card 1)
  - persists replacement on server so both players see the same replacement in PVP.
- Ensure “10-card match means 10 answers” invariant.

5) Masking (mission critical):
- Implement robust masking pipeline:
  - Always apply an overlay mask area at both top and bottom bands (configurable % of height).
  - Additionally, do client-side “dynamic mask” if OCR detects player name region (optional).
  - The safest baseline: mask BOTH the typical name locations (top banner + bottom nameplate) for all cards.
  - Store per-set/per-era mask presets if needed.
- Add automated visual tests using a sample set of known “name visible” cards.

6) Match state machine (PVP):
- Implement explicit match lifecycle in DB:
  - CREATED -> STARTED -> IN_PROGRESS -> COMPLETED -> CLOSED
- Per card (round) state:
  - round_index, card_id, answer_user1, answer_user2, locked_at, resolved_at
- Transition rules:
  - A user can submit once per round.
  - Round resolves when both answers present OR timeout triggers.
  - Only then advance to next round.
- Remove “waiting for opponent” deadlocks by:
  - subscribing to realtime updates (websocket/supabase) OR polling with exponential backoff
  - server authoritative advancement: clients request “next” and server returns current round state.

7) Authorization errors:
- Ensure all match routes require registered users; show login gate.
- Ensure API endpoints validate session token and user belongs to match.
- Fix any 401 causing “Not authorized” post-submit by ensuring auth cookies/headers included and server CORS correct.

Deliverable: /docs/GAMEPLAY_AUDIT.md with each historical bug class and the specific permanent fix.
