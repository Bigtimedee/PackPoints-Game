Implement a global “Card-Completion Reward Curve” that enforces this invariant across PackPTS:

INVARIANT (MUST HOLD EVERYWHERE)
A user can earn a MAX of 15,000 PackPTS for completing 200 total cards, regardless of how those cards are distributed across matches.
Example constraint: 10 matches × 20 cards per match = 200 cards total → max reward = 15,000 PackPTS.

The reward relationship must persist across the entire application (gameplay, streak bonuses exclusions, leaderboards, wallet ledger). This is a global cap+curve for “base gameplay earnings per card completion”.

========================================================
A) MATHEMATICAL RELATIONSHIP (GLOBAL)

Let:
- C = total number of cards the user has completed (correctly answered) in the current earning window
- C_max = 200
- P_max = 15,000 PackPTS

Define base earnings from card-completions as:
P(C) = floor( P_max * min(C, C_max) / C_max )

Which implies:
- Points per card (linear): P_per_card = P_max / C_max = 75 PackPTS per completed card
- Maximum: P(200) = 15,000
- Saturation: for C >= 200, P(C) = 15,000 (no more base earnings from cards)

Incremental points awarded on a correct card i:
deltaP = P(C_new) - P(C_old)
(where C_new = min(C_old + 1, 200))

This guarantees you never exceed 15,000 due to rounding or concurrency.

NOTE:
- “Match completion” is not directly rewarded. Only “card completion” (per correct card).
- This curve applies to BASE gameplay earnings. Other programs (streak/referral/admin promos) are separate ledgers and do not count toward this cap unless explicitly configured.

C) GLOBAL CONFIG (SINGLE SOURCE OF TRUTH)

Create:

export const GAMEPLAY_BASE_REWARD = {
  C_MAX: 200,
  P_MAX: 15000,
  pointsForCardsCompleted: (c: number) =>
    Math.floor(15000 * Math.min(c, 200) / 200)
};

Also export:

========================================================
D) AWARDING LOGIC (MUST BE ATOMIC + IDPOTENT)

Create:
src/services/rewards/gameplayBase.ts

Function:
awardForCorrectCard({ userId, matchId, cardId, windowStart }): Promise<{deltaPts:number, newTotalPts:number}>

Rules:
1) Idempotency:
- If gameplay_card_events has an entry for (userId, matchId, cardId) with is_correct=true already, return deltaPts=0.
2) Start a DB transaction:
- Insert gameplay_card_events (or check existing) to prevent double-award
- SELECT user_gameplay_counters FOR UPDATE
- If no row exists, create it with cards_completed=0, base_pts_awarded=0
- Compute:
  oldC = cards_completed
  newC = min(oldC + 1, 200)
  oldP = base_pts_awarded
  targetP = pointsForCardsCompleted(newC)
  delta = max(0, targetP - oldP)
- Update counters: cards_completed=newC, base_pts_awarded=oldP+delta
- Insert wallet_ledger entry EARN_GAMEPLAY_BASE with amount=delta (ONLY if delta>0)
- Commit
3) Return delta and new totals.

This guarantees:
- base_pts_awarded never exceeds 15,000
- concurrency safe
- points awarded exactly align with the mathematical curve
