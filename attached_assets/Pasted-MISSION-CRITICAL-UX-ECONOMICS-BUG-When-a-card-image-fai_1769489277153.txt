MISSION-CRITICAL UX/ECONOMICS BUG: When a card image fails to load, the “Skip to Next” button must NOT advance the match index (must NOT move the user from card 1 to card 2). Instead, it must trigger a “REPLACE THIS CARD” flow that swaps in a new card for the SAME question slot, so a 10-card match always results in 10 submitted answers.

After implementing, DO NOT test. User will test manually.

STACK: React 18 + Vite + TS + Express + ws + Postgres(Neon) + Drizzle + Passport session

========================================================
GOAL
- A 10-card game MUST always present 10 answerable questions.
- If an image fails to load, user can request another card for the SAME idx (question slot).
- Replacing a card must be server-authoritative (works for 1vComputer AND 1v1 matches).
- Limit abuse (rate limit + max replaces per idx).
- DO NOT increment idx until an answer is submitted (or a true “forfeit” flow you explicitly design later).

========================================================
A) DATA MODEL CHANGES (DRIZZLE + MIGRATION)

1) Add per-question “card assignment” table so idx maps to a cardId that can be replaced:

Table: match_questions
- match_id uuid (FK matches.id)
- idx int (0..questions_total-1)
- card_id text NOT NULL
- seed_version int NOT NULL DEFAULT 1      // increments on replace
- replaced_count int NOT NULL DEFAULT 0
- assigned_at timestamp NOT NULL DEFAULT now()
PRIMARY KEY (match_id, idx)

2) Add optional audit table:
match_question_events
- id uuid pk
- match_id uuid
- idx int
- user_id uuid
- event text  // 'ASSIGNED' | 'REPLACE_REQUEST' | 'REPLACED' | 'REPLACE_DENIED'
- meta jsonb
- created_at timestamp default now()

========================================================
B) SERVER: AUTHORITATIVE QUESTION PAYLOAD INCLUDES seedVersion

Whenever you send a question (QUESTION_NEXT or MATCH_STATE), include:
{ matchId, idx, card: { cardId, imageUrl, choices... }, seedVersion }

Clients must store seedVersion and send it back on replace requests to avoid races.

========================================================
C) SERVER: ADD “REPLACE CARD” ACTION (DO NOT ADVANCE IDX)

Implement WS event (and REST fallback):

Client -> Server:
QUESTION_REPLACE_REQUEST { matchId, idx, seedVersion, reason:"image_load_failed" }

Server rules (hard requirements):
1) requireAuth (ws.userId present)
2) assertParticipant(matchId, userId) for any 1v1 mode
3) Validate idx == matches.current_index (only replace current question)
4) Validate seedVersion matches current match_questions.seed_version
   - if mismatch: respond with MATCH_STATE (resync), do not replace
5) Abuse limits:
   - maxReplacesPerIdx = 3
   - cooldownSeconds = 3 between replaces
6) Choose a NEW card (not previously used in this match):
   - Maintain a set of used card_ids:
     Table: match_used_cards(match_id, card_id) PK(match_id, card_id)
   - When assigning/replacing, INSERT into match_used_cards; if conflict, pick another.
7) Transaction:
   BEGIN
   - Lock match row FOR UPDATE
   - Lock match_questions row FOR UPDATE
   - Check replaced_count < max
   - Generate replacement cardId
   - Update match_questions:
       card_id = newCardId
       seed_version = seed_version + 1
       replaced_count = replaced_count + 1
       assigned_at = now()
   COMMIT
8) Broadcast:
   - For 1vComputer: send to requester only
   - For 1vFriends / 1vRandom: broadcast to BOTH players in match room:
     QUESTION_REPLACED { matchId, idx, seedVersion:newSeedVersion, card: {...} }

IMPORTANT: This does NOT change matches.current_index and does NOT change progress (8/10 stays 8/10).

========================================================
D) CLIENT: CHANGE “SKIP TO NEXT” TO “TRY ANOTHER CARD (SAME SLOT)”

1) Rename button label (optional): “Try another card”
2) Current behavior (increment idx / local advance) must be removed.

In the card component:
- Detect image load failure via <img onError>
- When error occurs:
   - show error UI + button
- On button click:
   - call WS send QUESTION_REPLACE_REQUEST with (matchId, idx=currentIndex, seedVersion, reason)
   - show spinner “Fetching another card…”
   - when QUESTION_REPLACED arrives:
       - replace card payload in state
       - clear image error state
       - DO NOT change idx

3) Also implement auto-retry before showing button:
- Retry image fetch 2 times with cache-busting query param:
  imgSrc = `${imageUrl}${imageUrl.includes('?')?'&':'?'}t=${seedVersion}-${retryCount}-${Date.now()}`
- Only after retries fail show “Try another card”.

4) The progress indicator (e.g., “8/10”) must be derived from idx+1 (current card number) and questions_total, not “attempts”.
Replacing a card does not change idx.

========================================================
E) ANSWER SUBMISSION MUST REQUIRE CURRENT seedVersion

To avoid a race where a user answers an old replaced card:
Client -> Server ANSWER_SUBMIT includes seedVersion.
Server validates matches current match_questions.seed_version before accepting answer.
If mismatch, reject with STALE_QUESTION and send MATCH_STATE.

========================================================
F) KEEP “10 ANSWERS REQUIRED” INVARIANT

Server-side invariant:
- match ends only when idx reaches questions_total AFTER valid answers are submitted for each idx.
- Replacing a question never increments idx.
- Do not count “replace” as “attempt”.
