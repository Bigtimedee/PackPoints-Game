Implement a “Founders Cap” access system for PackPts that restricts activated users to a configurable limit (e.g., 500), with a waitlist + invite codes + admin overrides.

STACK (do not change)
- Frontend: React 18 + Vite + TS + Tailwind + shadcn/ui + React Query + Wouter
- Backend: Express + TS + Passport + express-session
- DB: Postgres (Replit Neon) + Drizzle ORM + migrations
- Existing auth: Replit OAuth + local username/password

GOALS
1) Cap ACTIVE registered users to MAX_ACTIVE_USERS (default 500).
2) After cap reached, prevent new account activation and route users to a waitlist flow.
3) Support invite codes that can bypass waitlist (optionally consuming reserved seats).
4) Admin dashboard/endpoints to manage cap, invites, and waitlist approvals.
5) Make it abuse-resistant: prevent a single person from taking multiple seats via email, device fingerprint, IP velocity limits.

NON-NEGOTIABLE CONSTRAINTS
- Cap enforcement must be server-side only.
- Cap must be enforced transactionally and atomically in DB to avoid race conditions.
- “Active user” means a user who completed registration and has status=ACTIVE (not WAITLISTED).
- Waitlist signups are allowed even after cap is reached.
- Provide feature flags / config in DB so we can adjust the cap without redeploying.

DATA MODEL (Drizzle migrations)
1) users table additions:
- status enum: 'PENDING' | 'ACTIVE' | 'WAITLISTED' | 'BANNED'
- activated_at timestamp nullable
- waitlist_joined_at timestamp nullable
- device_fingerprint text nullable (hashed)
- last_signup_ip text nullable
- email_normalized text (lowercased) and enforce uniqueness if possible

2) app_config table (if not existing):
- key text primary key
- value jsonb
Seed keys:
- 'founders_cap' = { "maxActiveUsers": 500, "enabled": true, "inviteBypass": true, "reservedSeatsForInvites": 100 }
- 'waitlist' = { "enabled": true }

3) waitlist_entries table:
- id pk
- email text
- email_normalized text
- name text nullable
- created_at timestamp
- status enum: 'WAITING'|'INVITED'|'ACCEPTED'|'REJECTED'
- position int (assigned at join time)
- referral_code text unique nullable
- referred_by_code text nullable
- referrals_count int default 0
Indexes: (email_normalized), (status), (position)

4) invite_codes table:
- code text primary key (short)
- created_at timestamp
- expires_at timestamp nullable
- max_uses int default 1
- uses int default 0
- reserved_seat boolean default true
- created_by_admin_user_id nullable
- note text nullable

5) audit table:
- access_audit_log: id, action, user_id/email, ip, user_agent, metadata jsonb, created_at

BACKEND LOGIC
A) Core cap check (atomic)
Implement a function `tryActivateUser(userId, context)` that:
- Reads founders_cap config (enabled, maxActiveUsers, reservedSeatsForInvites, inviteBypass)
- In a DB transaction with appropriate locking:
  1) Count ACTIVE users (or maintain a counter row and lock it)
  2) If ACTIVE >= maxActiveUsers:
       - set user.status='WAITLISTED', set waitlist_joined_at
       - return { activated:false, reason:'CAP_REACHED' }
     Else:
       - set user.status='ACTIVE', set activated_at=now()
       - return { activated:true }

IMPORTANT: Ensure no race condition allows > maxActiveUsers ACTIVE users.

B) Invite bypass flow
Implement:
- POST /api/access/validate-invite { code }
  - checks validity, expiry, remaining uses
- POST /api/access/consume-invite { code }
  - increments uses atomically
  - marks session as inviteApproved = true (req.session.inviteApprovedCode = code)
- Modify activation flow:
  - If inviteApprovedCode exists AND inviteBypass enabled:
      - allow activation even if cap reached IF either:
        a) reservedSeatsForInvites not exceeded (track reserved seat usage), OR
        b) reserved_seat=false for code (strictly bypass cap)
  - Record audit log.

C) Waitlist join + position
- POST /api/waitlist/join { email, name?, referredByCode? }
  - normalizes email, checks if already exists
  - creates waitlist entry with next position (max(position)+1)
  - generates referral_code
  - increments referrer’s referrals_count if referredByCode provided
- GET /api/waitlist/status?email=...
  - returns position, status, and whether invites are available

D) “Activation gate” integration
Integrate with existing auth flows:
- After successful authentication (local or Replit OAuth), call `tryActivateUser`.
- If activated:false:
   - log user out OR keep a limited session but block app access
   - redirect frontend to /waitlist (show their position)
- If activated:true:
   - proceed normally.

E) Middleware
- requireActiveUser middleware for all gameplay and store endpoints:
  - if req.user.status != 'ACTIVE' => 403 with code WAITLISTED

F) Admin endpoints (protected by admin role)
- GET /api/admin/access/summary
  - activeCount, cap, reserved seats usage, waitlist size
- POST /api/admin/access/cap
  - update founders_cap config (maxActiveUsers, enabled, reservedSeatsForInvites)
- POST /api/admin/invites/create
  - create invite codes (bulk) with max_uses, expires_at, reserved_seat
- POST /api/admin/waitlist/invite
  - mark waitlist entry as INVITED and optionally email them an invite code (reuse Nodemailer)
- POST /api/admin/waitlist/approve
  - move WAITLISTED user to ACTIVE if cap allows OR by consuming reserved invite seat

FRONTEND (React)
Routes:
- /waitlist
  - join waitlist form (email)
  - if logged in but WAITLISTED, show position + referral link
- /invite
  - input invite code -> validate -> consume -> then prompt sign-in
- Update login/signup page:
  - show “Founders: Limited to 500” with remaining seats (fetch /api/access/summary public version)
  - show “Have an invite code?” link to /invite

ABUSE CONTROLS
- Normalize emails and enforce uniqueness.
- Store a hashed device fingerprint (client-generated) and rate limit signups per device/ip/day.
- Prevent multiple user accounts from claiming seats with same device fingerprint.
- Add velocity limits and audit logs.
